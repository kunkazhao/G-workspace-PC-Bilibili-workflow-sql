## 2. 🔴 P0 — 严重问题

| 编号 | 位置 | 问题 | 影响 | 修复建议 |
|---|---|---|---|---|
| **P0-1** | `bworkflow_sql/utils.py:22` + `bworkflow_sql/db.py:198-204` | `text_hash()` 返回 SHA1(40 字符十六进制),但 `_migrate_script_hashes` 用 `len(current) != 64` 判断是否要重算。**所有现存 hash 都是 40 字符,会被判定为"已正确"而跳过迁移**;一旦将来切到 SHA256(64 字符),新行用新算法、老行用旧算法,跨行一致性破坏。 | 配音去重失效、口播稿拼接错乱、跨账号 hash 不可比 | 立即统一 hash 长度(推荐 SHA256),迁移判断改为 `if not current or len(current) != 64` |
| **P0-2** | `bworkflow_sql/sync_service.py:163` vs `bworkflow_sql/db.py:222-224` | intro 的 `script_id` 命名空间分裂:`sync_markdown_payload` 写入 `intro:V001`,`_migrate_script_ids` 写入 `intro:I001`。**两个入口写不同前缀**,导致配音匹配、口播稿引用、字幕锚点对不上。 | 配音找不到、文案块孤儿、字幕错位 | 统一为 `intro:I{NNN}`,并写一次性 `UPDATE` 把已存在的 `intro:V*` 改成 `intro:I*` |
| **P0-3** | `bworkflow_sql/workflow_service.py:1316-1332` | roll-b 改名的回滚逻辑有漏洞:line 1319 `source.rename(temp)` **无 try/except**,失败后 line 1320 `temp_pairs.append(...)` 仍会执行;line 1321 `temp.rename(target)` 抛 `FileNotFoundError`,**触发 except 后回滚不存在的 temp**(`if temp.exists()` 保护了,但等于"完全不恢复"且用户以为安全)。 | 视频文件可能丢失、改名回滚不完全 | `rename` 前加 `if not source.exists()` 守卫;`temp_pairs.append` 只在 `rename` 成功后才执行;整段包 `try/except/finally` |
| **P0-4** | `bworkflow_sql/workflow_service.py:1479` + `:1557` | 多次配音同一文案时 `unique_path()` 返回 `filename-1.wav` / `filename-2.wav`... 数据库 `asset_bindings.path` 累积更新成越来越长的文件名;**旧的 ready 记录 path 指向 `filename.wav` 但磁盘上是 `filename-3.wav`,永远找不到对应文件**。 | 数据库膨胀、配音状态永远显示"未生成"、旧文件成孤儿 | 改用稳定文件名(`{uid}-{script_id}-{hash8}.wav` 等),文件名不依赖生成次数 |
| **P0-5** | `bworkflow_sql/sync_service.py:230-305` | `_write_script_ids_to_markdown` 直接 `path.write_text(...)` 覆盖用户 MD,**没有原子性**;进程崩溃/断电会留下半截文件或空文件;**且修改了用户原文档但没有备份**。 | 用户原始文案可能丢失 | 写到同目录 `tmp` 文件,`os.replace(tmp, original)`;写之前复制一份 `.bak` 备份 |

## 3. 🟠 P1 — 高风险问题

| 编号 | 位置 | 问题 | 影响 | 修复建议 |
|---|---|---|---|---|
| **P1-1** | `bworkflow_sql/sync_service.py:466-480` | `UPDATE asset_bindings SET status='stale' WHERE id IN ({placeholders})`,当 `removed_ids` 数量超过 999 时触发 SQLite 默认 `SQLITE_MAX_VARIABLE_NUMBER`,**SQL 构造失败抛错,sync 中断**。 | 大型项目 sync 失败 | 分批执行(每批 500 条),或写入临时表后 `JOIN` 更新 |
| **P1-2** | `bworkflow_sql/legacy_import.py:311-320` | `_prune_out_of_scope_assets` 同样的 `IN (...)` 拼 SQL 模式,`allowed_uids` 超过 999 报错。 | 旧项目导入失败 | 同 P1-1,分批执行 |
| **P1-3** | `bworkflow_sql/sync_service.py:378-394` | 文件扫描 + UID 匹配 + 账号匹配三层 O(N) 循环:每个文件做 N_uid 次正则 + N_account × N_keys 次子串匹配。**1000 文件 × 50 UID × 10 账号 = 几十万次字符串操作**。 | 大型项目同步卡顿 5-30 秒 | 启动时预编译所有 UID 正则 + 账号 key 小写集合,每个文件只跑一次 |
| **P1-4** | `bworkflow_sql/sync_service.py:638-647` `_uid_from_path` | 每次调用都 `re.escape(uid_text) + UID_BOUNDARY_PATTERN.format(...)` 然后 `re.search`,**没有预编译**。 | 性能差 + 维护性差 | 在 `__init__` 阶段 `re.compile` 所有 UID 正则并存入缓存 |
| **P1-5** | `bworkflow_sql/workflow_service.py:131-191` `dbfs_for_chunk` / `silence_ranges_for_audio` | 把整个音频缓冲区当成单通道流处理,**未区分立体声**。立体声 WAV 的左右声道被合并为一个流计算静音,dbfs 不准确。 | 立体声音频的静音修音效果可能偏差(本项目多数配音是单声道,实际影响小) | 按 channel 分别处理 dbfs,或至少加注释明确"按单声道优化" |
| **P1-6** | `bworkflow_sql/workflow_service.py:1316-1318` | `temp = ...-{index}{suffix}` 然后 `while temp.exists(): temp = ...-{index+1}{suffix}`。**只递增 index 但不更新时间戳**,且 index+1 可能被前一轮占用。**1 秒内同一源文件反复重命名可能死循环**。 | 极端情况死循环 | 加最大尝试次数限制(如 1000 次)或改用 UUID |
| **P1-7** | `bworkflow_sql/workflow_service.py:1660-1678` `_launch_tts_api` | `stdout_log` / `stderr_log` 文件句柄存到 `self._tts_log_handles`,**每次启动重置列表 = 旧句柄泄漏**;`CREATE_NEW_PROCESS_GROUP` 让子进程与父进程解耦,父进程退出不会带走子进程。 | 句柄泄漏 + 僵尸 TTS 进程 | 启动前先 close 旧句柄;定期清理 dead TTS 进程(pid 文件 + 探活) |
| **P1-8** | `bworkflow_sql/workflow_service.py:1681-1702` `_find_tts_service_pids` | 用 `f":{port}" not in line` 行匹配,可能误匹配其他含相同端口的输出行(如 UDP/连接中行);**且没区分 IPv4/IPv6 监听**。 | 误杀其他进程 / 找不到 TTS PID | 用 `line.split()` 解析列,只取 TCP / LISTENING / IPv4 行 |

## 7. 附:审查时顺手发现的非问题(确认 OK)

- ✅ `Database.connect()` 使用 `with sqlite3.connect(...)` 模式,资源管理 OK
- ✅ `_migrate()` 用 `if column not in columns` 守卫,不会重复 ADD COLUMN
- ✅ schema 启用 `PRAGMA foreign_keys = ON`,外键级联删除 OK
- ✅ `safe_text()` / `now_iso()` 等工具函数简单可靠
- ✅ UI 进度对话框用 `transient()` 但显式避开已知 bug,有注释说明
- ✅ ASR 字幕对齐用 `ProcessPoolExecutor` 隔离 CPU 密集任务,设计合理
- ✅ 静音检测对单声道配音场景(本项目主流)是准确的
